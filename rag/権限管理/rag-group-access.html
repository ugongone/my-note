営業部・開発部の両方のメンバーが閲覧できる

2データの紐付け構造

3つの多対多関係

ユーザー
田中、佐藤、山田...
所属（N : N）
グループ
営業部、開発部、経営層...
公開設定（N : N）
ドキュメント
営業マニュアル、API仕様書...
関係 意味 例
ユーザー × グループ 誰がどのグループに所属するか 佐藤 → 営業部, 開発部
グループ × ドキュメント どのグループにドキュメントを公開するか 営業マニュアル → 営業部
公開フラグ（public）: ドキュメントに public: true を設定すると、グループ紐付けに関係なく全員に公開される。

3検索ヒット判定ロジック

ドキュメントが検索にヒットする条件

// ドキュメントが検索結果に含まれる条件
IF ドキュメント.public == true
    → 全員にヒット（グループ不問）

OR

IF ドキュメントの公開先グループ ∩ ユーザーの所属グループ ≠ 空
    → 1つでも一致するグループがあればヒット

5具体例：誰に何が見えるか

佐藤さん（営業部 + 開発部 所属）が検索した場合

ドキュメント 結果 理由
営業マニュアル ✅ ヒット 営業部メンバーだから
価格表 ✅ ヒット 営業部メンバーだから
API仕様書 ✅ ヒット 開発部メンバーだから
設計書 ✅ ヒット 開発部メンバーだから
就業規則 ✅ ヒット public: true（全員公開）
経営会議議事録 ❌ 見えない 経営層グループに未所属

山田さん（開発部のみ）が同じ検索をした場合

ドキュメント 結果 理由
営業マニュアル ❌ 見えない 営業部に未所属
API仕様書 ✅ ヒット 開発部メンバーだから
設計書 ✅ ヒット 開発部メンバーだから
就業規則 ✅ ヒット public: true（全員公開）
経営会議議事録 ❌ 見えない 経営層グループに未所属
