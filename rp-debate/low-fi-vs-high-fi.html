Lo-Fi vs Hi-Fi

RPにおけるプロトタイピング手法の論争を、AIエージェント同士に議論させて検証した

背景

Rapid Prototyping（RP）とは

ソフトウェア開発において、完成品を作る前に「簡易的な試作品（プロトタイプ）を素早く作り、クライアントや関係者にフィードバックをもらう」プロセスのこと。要件の認識ズレや方向性のミスを早期に発見し、手戻りを最小化するために行う。

Lo-Fi（低忠実度）— ワイヤーフレーム例

ログイン

ワイヤーフレームや手書きスケッチ。見た目の完成度は低いが、素早く作れて変更しやすい。

Hi-Fi（高忠実度）— 画面イメージ例

ログイン

実際の画面に近いデザインや動作するHTML。見た目・操作感まで検証できる。

このディベートの問い

AIによりHi-Fiの作成コストが激減した今、どちらから始めるのが最適か？

AIによるHi-Fiの高速化

昨今のAIの進化により、完成物に近いHTMLを数分で生成できるようになった。ワイヤーフレーム（Lo-Fi）との作成工数差はほぼゼロになりつつある。

残る問い

一方で「RPではLo-Fiなワイヤーフレームをクライアントに見せる方が良い」というアイデアもある。工数差がなくなった今、それでもLo-Fiを選ぶ理由はあるのか？

この問いを解決するため、Claude Code の Agent Teams 機能を使い、AIエージェント同士にディベートさせることにした。

ディベートの設計

使用機能：Claude Code — Agent Teams

Agent Teams は Claude Code のチーム機能で、複数のAIエージェントを並列に起動し、メッセージを送り合いながら協調・議論させることができる。今回はこれを使い、対立する立場を持つ2エージェントに非同期でターン制のディベートを行わせた。

各エージェントは独立したコンテキストを持ち、team-leadがメッセージを転送・仲裁する形で議論を進行。

A

agent-A

Lo-Fi 支持

ワイヤーフレームなどのLo-Fi案を支持。機能・フロー議論への集中、方向転換コストの低さを主軸に論じる。

B

agent-B

Hi-Fi 支持

AIで完成物HTMLをいきなり作る案を支持。クライアントへの具体性・成果物の資産性・AIによるコスト革命を主軸に論じる。

team-lead

進行・ジャッジ担当エージェント

各ターンの主張を相手に転送し、争点が揺れる論点についてユーザーに適宜質問しながら議論を深める。最終的にジャッジを下す。

進行上の工夫と注意点

各ターン終了後にユーザーへ質問：争点が揺れそうな論点（「実際のセッション形式は？」「big pivotはどちらで起きやすいと思うか？」など）をターンごとにユーザーに確認し、回答を次のターンに反映した。ユーザーにとって重要でない部分で論争が長引くのを防ぐための工夫。
注意：ユーザーのバイアスが入っている：質問への回答はユーザーの主観・経験・期待に基づくものであり、議論の方向がその回答に引き寄せられている。
注意：結論はこのコンテキスト限定：前提条件とQAによってピンポイント化された議論であるため、チーム構成・クライアントのリテラシー・開発フェーズが異なるケースにそのまま適用できるわけではない。

与えた情報

1 最初から与えた前提条件

対象シナリオ

社内の特定チームがクライアント。PoCをアジャイルで実施し、高速に動くアプリを提供したい。

工数差

AIの進化により、ワイヤーフレームとHi-Fi HTMLの作成工数差はほぼゼロ。紙スケッチの方がむしろ作りづらい。

変更への抵抗

AIに直させるだけなのでHi-Fiを壊すことへの抵抗が全くない。

PoCの目的

機能の優先順位付けに特化。細かいUXはこの時点で不要。

クライアント

UXデザインに精通していない。ステークホルダーは少人数。

方向性

何を作るかまだ未確定。RPの範囲（PoC部分のみかアイデア全体か）も未決定。

ディベートの様子（抜粋）

各エージェントのメッセージは非同期で届き、team-leadが転送・仲裁しながら議論を進めた。

agent-A の Opening Statement（Lo-Fi支持）

## Opening Statement - agent-A（Lo-Fi案支持） ### 1. フィードバックループの質と速度 Lo-Fiでは、見た目の完成度が低いため、クライアントは UIの細部（色・フォント）ではなく機能・フロー・情報設計に 集中してフィードバックできます。Hi-FiのHTMLを最初に 見せると「このボタンの色が気になる」といった表面的な意見が 増え、本質的な要件の検証が後回しになりがちです。 ### 2. 変更コストの圧倒的な低さ PoCは仮説検証の場であり、方向転換が頻繁に起きます。 ワイヤーフレームであれば、ページ構成の変更・機能の追加削除を ほぼゼロコストで行えます。 ### 3. 関係者間の共通理解の形成 ワイヤーフレームは「まだ決まっていないものを議論するための道具」 として機能します。「これはたたき台だ」と共通認識を持った状態で 議論できることが、後続のスプリントの質を左右します。

agent-B の反論（Hi-Fi支持・ターン1）

### 反論2：AIによる修正コストはLo-Fiと変わらないレベルまで下がっている 「Hi-Fiは変更コストが高い」という主張は、AIが登場する前の 世界の常識だ。現在はAIに「このページのナビゲーションを削除して、 フローをA→B→Cに変えて」と指示すれば数分で修正できる。 そして決定的な点として、Lo-Fiで検証した後にHi-Fiを作る工程が 丸ごと発生する。これは「変更コストの削減」ではなく「工程の追加」だ。 Hi-Fiで始めれば、この二重投資が不要になる。 ### 反論3：「もうほぼできている」という誤解は、資産になる PoCの現実には「あれもこれも」というスコープクリープがある。 Hi-Fiで「この範囲でこれだけのものが作れる」と具体的に示すことで、 議論を現実的な範囲に収束させる効果がある。

ターン2終了後の争点整理（team-lead）

## ターン2終了後の注目点 「Lo-FiかHi-Fiか」という二項対立が崩れ始めています。 agent-Bが「AI生成時にグレースケールを指定することで 視覚的ノイズを排除できる」と主張。 クライアントからは「良いアイデアだ（中間状態として有効）」という 回答が得られました。 これにより agent-A は「それはもはやLo-Fiと同じではないか」 という反論を使える状況に。 また「big pivot はどちらでも起きうる」という確認から、 両者の核心的な争点が崩れ、「何をLo-Fiと呼ぶかの定義争い」 にシフトしてきました。

　判定と結論

判定：Hi-Fi（agent-B）優位

決め手：成果物の資産性
工数差・変更コスト・big pivot・認知バイアスのほぼ全論点が「引き分け」に収束した結果、「作るたびに資産が積み上がるHi-Fi HTML」という唯一の差分が決め手となった。同じ議論品質が得られるなら、次のイテレーションで直接使える成果物を選ぶ方が合理的。

ディベートから得た教訓

問いの変換：「Lo-FiかHi-Fiか」は二項対立ではなく、「どの視覚的完成度で始めるか」というスペクトラムの問題だった。
コンテキスト依存：工数差・変更コスト・チーム文化・クライアントのリテラシーによって最適解は変わる。AIで工数差が消えた現在、具体的なコンテキストを与えるほどHi-Fiが有利になる傾向がある。
Agent Teamsの有効性：各ターンでクライアントに質問しながら情報を補完していくプロセスが、議論の解像度を上げた。単なる一問一答より深い検討ができた。
