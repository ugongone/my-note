Power Automate Desktop vs Playwright

アーキテクチャ・DOM操作・実行環境・AI統合の技術的差異を徹底分析

はじめに

Power Automate Desktop（PAD）とPlaywrightは、どちらもWebブラウザの操作を自動化できるツールですが、内部アーキテクチャは根本的に異なります。本記事では、表面的な機能比較ではなく、なぜ安定性に差が出るのか、なぜAIとの親和性が異なるのかを技術的な構造から解説します。

対象読者: ブラウザ自動化の技術選定を行うエンジニア、RPA導入を検討中のテックリード、AIエージェント開発者

この構造差がもたらす具体的な影響

レイテンシ

PADは多層構造で遅延が増大。Playwrightは直接制御で低レイテンシ

安定性

PADは拡張機能更新で動作不良のリスク。Playwrightはバージョン固定で安定

JS実行

PADはWindows JScript経由。PlaywrightはブラウザV8エンジンで直接実行

DOM操作能力の実力差

セレクター方式の違い

HTML上の要素を「どうやって見つけるか」の仕組み。セレクターが壊れると自動化は即停止する

PAD：独自階層型セレクター

// PADの独自セレクター形式
element[id="dynamic-123"] > div[class="css-abc"] > button
  • -Equal To / Contains / Regex等の演算子
  • -複数セレクターのフォールバック登録可能
  • -レコーダーが動的ID・ハッシュクラスを含むセレクターを生成しがち
  • -テキストベース選択はUIピッカーで明示的に選んだ場合のみ

Playwright：セマンティックロケーター

// 意味ベースの安定した要素特定
page.getByRole('button', { name: '検索' })
page.getByText('ログイン')
page.getByTestId('submit-btn')
  • -CSS / XPath / テキスト / ロール / data-testid
  • -IDに依存しない「意味ベース」の要素特定
  • ->> 演算子でShadow DOMを貫通
  • -DOM構造の変更に強い

JavaScript実行の仕組み

ブラウザ内でJavaScriptを実行し、ページの値取得や操作を行う方法の違い

PAD

Run JS アクション
Windows JScript
拡張機能ブリッジ
ブラウザ

Windows JScript（旧WScript）で実行後、拡張機能経由でブラウザに注入。レイテンシが高く確実性が低い

Playwright

page.evaluate()
V8 / SpiderMonkey

ブラウザ内部エンジン（V8等）で直接実行。DOM操作が低レイテンシで確実。ネットワーク傍受・モックも可能

自動待機（Auto-waiting）

ページの読み込みや要素の表示を「待ってから操作する」機能。これがないと動的なページで頻繁にエラーが起きる

PAD 手動対応が必要

  • -自動待機機能なし
  • -「Wait for web page content」アクションで対応
  • -動的ページでは不十分な場合が多い
  • -タイミング依存のエラーが頻発

Playwright エンジンレベル対応

  • -Auto-waiting標準搭載
  • -DOM存在・表示・安定・操作可能を自動検知
  • -waitForLoadState / waitForResponse等で細かい制御
  • -SPA非同期レンダリングにエンジンレベルで対応

Shadow DOM / iframe / SPAでの成功率

モダンWebで多用される複雑なページ構造への対応力。業務SaaSの多くがこれらを利用している

対象 PAD Playwright
Shadow DOM 2025年GA（新） 初期から透過対応
iframe（同一ドメイン） ○ 対応 ◎ frameLocator()
iframe（クロスドメイン） △ 回避策が必要 ◎ 統一API
静的HTML 90%+ 95%+
SPA（React/Angular/Vue） 50-70% 85-95%
PADがSPAで失敗する主因: Reactの再レンダリングでDOM要素が破棄→再生成されると要素参照が失われる。ハッシュ付きクラス名（css-1a2b3c）がビルドごとに変わりセレクターが壊れる。

マルチタブ操作

PAD：アクティブタブ依存

  • -前面タブのみ操作可能
  • -バックグラウンドタブは操作不可
  • -30タブ以上でエラー増加
  • -動的タブはAttach to running instanceで再接続が必要
  • -タブ切り替え時にフォーカス確認が必要

Playwright：Pageオブジェクト管理

  • -各タブが独立したPageオブジェクト
  • -バックグラウンドでも操作可能
  • -何個でも安定管理（配列でスケール）
  • -イベント駆動の精密な待機API
  • -タブ間の行き来が変数参照で自然

典型パターン：検索結果→詳細タブ→戻る

// Playwright: タブ間のデータ連携が自然
const results = await pageA.$$('a.result-link');
for (const link of results) {
  const [pageB] = await Promise.all([
    context.waitForEvent('page'),
    link.click(),
  ]);
  await pageB.waitForLoadState();
  const detail = await pageB.locator('.detail').textContent();
  data.push(detail);
  await pageB.close(); // pageAは変数で保持したまま
}

PADでは同等の処理に、タブ切り替え→フォーカス確認→Attach→データ取得→Close→再Attach のフローが必要で、タイミングずれのリスクが高い

まとめ：どう使い分けるか

Playwrightを選ぶとき

  • Webアプリ/SaaSの自動化が中心
  • SPA（React/Angular/Vue）がターゲット
  • マルチタブ操作やタブ間データ連携が必要
  • AIエージェント統合を前提とする
  • Docker / Kubernetes上で実行したい

PADを選ぶとき

  • デスクトップアプリ（Excel、SAP等）を含む自動化
  • エンジニアリソースがないチーム
  • M365 / Power Platformエコシステムを活用中
  • ガバナンス・コンプライアンスが最優先
  • 操作が極めて単純（数クリック程度）

PADとPlaywrightの差は「機能の多寡」ではなく、アーキテクチャの根本的な設計思想の違いに起因します。PADは「GUI操作の再現」を設計原理とし、Playwrightは「ブラウザエンジンの直接制御」を設計原理としています。

この構造差が、DOM操作の安定性、マルチタブの扱い、実行環境の柔軟性、そしてAIとの統合のしやすさすべてに波及しています。どちらが「優れている」かではなく、自分のユースケースに合った設計思想のツールを選ぶことが重要です。

2026年のトレンド: AIエージェントがブラウザを操作する時代において、Playwrightはその基盤技術としてデファクトスタンダードの地位を確立しています。AIコーディング・AIエージェント統合を視野に入れるなら、Playwrightを基盤にすることが現時点での最適解です。
